# ADR-0021: Agent-Driven Memory Ownership

**状态**:Accepted
**日期**:2026-05-15
**决策者**:Wang
**相关 Round**:Round 75

---

## 背景与问题

ADR-0020 已决定 Phase 7 第一版共享记忆使用 AICO 本地可审计记忆账本,并提供
`/remember`、`/recall`、`/forget` 命令。

新的产品边界是:AI Company OS 的老板是来管理一个小中型项目和自己的 agent 团队的,
不是来维护记忆数据库的。如果记忆层要求老板高频手动执行 `/remember`、`/recall`、
`/forget`,会让记忆从提效能力变成焦虑来源。

## 候选方案

### 方案 A — 老板手动维护记忆
- 优点:实现简单;每条记忆都来自显式命令。
- 缺点:老板需要不断判断什么该记、什么时候查、何时忘;认知负担高,容易害怕“没记全”或“记错了”。
- 复杂度:低。

### 方案 B — 完全自动记住一切
- 优点:老板几乎不用操作。
- 缺点:容易把噪声、临时想法和错误输出写成长记忆;后续纠错困难,也违反可审计边界。
- 复杂度:中。

### 方案 C — Agent 主动维护,命令作为纠错和排障入口
- 优点:把日常记忆负担交给 agent;老板仍可查看、修正、忘记错误事实;符合“管理团队”而不是“维护数据库”的产品定位。
- 缺点:需要定义 agent 何时写入、召回和归档的规则,并在审计日志中保留理由。
- 复杂度:中。

## 决策

选择 **方案 C:Agent 主动维护,命令作为纠错和排障入口**。

Phase 7 第一切片保留 `/remember`、`/recall`、`/forget`,但它们不是老板的主工作流:
- 老板主体验仍是 `/project`、`/team`、`/ask`、`/brief`、`/daily`、`/weekly` 等项目管理命令。
- Agent 在完成任务、交接、发现稳定项目事实、识别风险或生成报告时,可以主动写入或更新记忆。
- Agent 在接到项目任务前,默认从当前项目召回少量高置信记忆,不要求老板先 `/recall`。
- 老板使用 `/remember` 主要用于纠正或补充关键事实。
- 老板使用 `/recall` 主要用于检查 agent 为什么这么做,或排查记忆污染。
- 老板使用 `/forget` 主要用于归档错误、过期或不应继续影响项目的记忆。

## 决策理由

- 符合北极星第一句:老板要像管理真实团队一样管理 AI,真实团队会主动做会议纪要、交接和背景同步,不会要求老板维护每个人的记忆。
- 符合北极星第二句:记忆仍是可插拔能力和共享项目状态,但触发层应该服务于 agent 协作,不是把命令负担暴露给人。
- 符合北极星第三句:自动写入必须可审计、可纠错、可归档;命令存在是为了控制权,不是为了把操作成本转嫁给老板。

## 后果

### 正面后果
- Phase 7 的产品体感会更接近“项目办公室自动沉淀上下文”。
- 老板不需要学习和维护一组高频记忆命令。
- 后续 agent 交接、日报、周报、风险报告可以自然使用共享记忆。

### 负面后果
- 第一版实现需要区分“agent 写入理由”和“老板手动写入”。
- 自动写入策略必须保守,否则容易把临时信息固化。
- 测试不只要覆盖命令,还要覆盖 agent 任务链路中的记忆读写。

### 我们接受这些代价是因为

Phase 7 的价值不是多几个 slash command,而是让老板少操心、让 agent 更像一个会交接和沉淀上下文的团队。

## 不再做的事

- 不把 `/remember`、`/recall`、`/forget` 设计成老板高频主路径。
- 不要求老板为每个任务手动选择要召回哪些记忆。
- 不把“显式命令写入”当成 Phase 7 记忆层的唯一入口。
- 不自动记住所有聊天和模型输出;agent 写入仍要有来源、置信度和可审计理由。
