# ADR-0005: Phase 4 审批与审计边界

**状态**:Accepted
**日期**:2026-04-28
**决策者**:Codex
**相关 Round**:Round 13

---

## 背景与问题

Phase 1-3 已经让 Telegram 可以远程派发任务、查询状态、点名 Adapter 和 broadcast 多 Persona。下一步风险是:远程 IM 一句话可能触发写文件、执行命令或破坏性操作。如果默认放行,会违反北极星第三句的“可审批、可审计、可中断”。

本阶段需要先建立最小审批和审计边界,让危险任务不会直接派发给 Adapter,同时让关键动作有结构化 trace。

## 候选方案

### 方案 A — 继续依赖 Adapter 自身安全模式
- 优点:无需改编排核心。
- 缺点:不同 CLI 安全语义不一致,核心无法统一观察和审批。
- 复杂度:低。

### 方案 B — TaskBus 前置风险识别 + Telegram 手动审批 + 内存审计
- 优点:最小改动即可覆盖远程 IM 风险入口,复用现有任务状态和 `/status`。
- 缺点:风险识别仍是文本启发式,审计暂不持久化。
- 复杂度:中。

### 方案 C — 完整审批工作流引擎 + 持久化数据库
- 优点:可扩展到多审批人、超时、审计查询和看板。
- 缺点:Phase 4 起步过重,会提前引入数据库和工作流复杂度。
- 复杂度:高。

## 决策

选择 **方案 B — TaskBus 前置风险识别 + Telegram 手动审批 + 内存审计**。

## 决策理由

- 风险入口在 `TaskBus.submit()` 前置拦截,不需要修改每个 Adapter。
- `waiting_approval` 成为任务生命周期的一等状态,可被 `/status` 直接观察。
- `/approve <task_id>` 和 `/reject <task_id>` 足够支持第一轮 Telegram dogfooding。
- 内存审计事件先记录任务提交、审批请求、审批结果、Adapter 派发和任务完成/失败/中断,后续可替换为日志或仓储 sink。
- 不引入新依赖,避免 Phase 4 一开始就变成平台化工程。

## 后果

### 正面后果
- 写文件、shell 执行、破坏性任务不会直接派发。
- 危险任务可以被人工批准后继续执行。
- 任务状态和审计事件模型开始进入核心协议。

### 负面后果
- 当前风险识别是文本启发式,会有误报和漏报。
- 审计事件暂存在内存里,进程重启会丢失。
- 当前任何 Telegram 用户知道 task id 都可审批,尚未做审批人权限策略。

### 我们接受这些代价是因为
- Phase 4 第一目标是阻止远程 IM 默认 YOLO,不是一次性完成合规平台。
- 持久化、审批权限和更准确的 tool-call 级风险识别可以在真实 smoke test 后迭代。

## 不再做的事

- 本阶段起步不引入数据库或工作流引擎。
- 不把审批逻辑散落到各 Adapter 内部。
- 不把 Codex / Claude 某个 CLI 的安全参数当作唯一防线。
